# 🧭 Отчёт о проделанной работе: Django REST Framework + React (JWT и Email Login)

## 1. Переход от классического Django к Django REST Framework (DRF)

### 🔹 Что было

Ты начал с проекта Django, где регистрация и авторизация были реализованы обычным путём:
через представления (`views.py`) и шаблоны (`templates`), с формами, страницами и редиректами.
Это классический способ для серверных приложений, где **Django рендерит HTML** и полностью управляет страницами.

### 🔹 Почему этого недостаточно

Такой подход неудобен, когда фронтенд написан отдельно — например, на **React**.
React работает как самостоятельное приложение, которое получает данные **через API**, а не через HTML-страницы.
Поэтому нам нужно было превратить Django из “рендерера страниц” в **REST API**,
которое возвращает **JSON**, а не HTML.

---

## 2. Подключение и настройка DRF

### 🔹 Что мы сделали

Мы установили библиотеку **Django REST Framework (DRF)**.
Она добавила возможность писать API-эндпоинты, сериализовать данные и управлять доступом через токены.

### 🔹 Теория

DRF превращает Django во **фреймворк для REST API**, добавляя три ключевые концепции:

* **Serializers** — преобразуют объекты Django (модели) в JSON (и обратно).
* **Views / ViewSets** — логика, которая обрабатывает HTTP-запросы (`GET`, `POST`, `PUT`, `DELETE`).
* **Permissions** — система прав доступа, например, кто может вызывать эндпоинт.

С этого момента Django перестаёт “рисовать страницы” и начинает “отдавать данные”.

---

## 3. Первые API: регистрация и авторизация (Token-based)

### 🔹 Переход от `@api_view` к `APIView`

DRF позволяет писать API двумя способами:

1. **@api_view** — простой функциональный подход, похожий на обычные Django views.
2. **APIView** — классовый подход, более гибкий и масштабируемый.

Мы выбрали **классовый подход** (через `APIView`), потому что он:

* лучше подходит для больших проектов;
* легко расширяется;
* поддерживает переопределение методов (`get`, `post`, `put`, и т.д.);
* используется в большинстве production-проектов.

---

## 4. Аутентификация через TokenAuthentication

### 🔹 Что такое токен

Токен — это длинная строка (например, `ac23a563f9ac31e2ec417915ae8e1db00aab46cb`),
которую сервер выдаёт пользователю после логина.
Дальше этот токен отправляется в каждом запросе в заголовке:

```
Authorization: Token <токен>
```

Он заменяет сессии и cookies, что делает API независимым от браузера.

### 🔹 Как это работает

1. Пользователь логинится (`POST /login`).
2. Сервер проверяет данные и выдаёт токен.
3. Токен хранится на фронтенде (в `localStorage`).
4. При каждом запросе React отправляет этот токен.
5. DRF проверяет его через `TokenAuthentication`.

---

## 5. Переход к JWT (JSON Web Token)

### 🔹 Почему JWT лучше

**JWT (JSON Web Token)** — это современная альтернатива классическим токенам.
Вместо хранения токенов на сервере (в базе), JWT — **самодостаточный токен**, который содержит:

* id пользователя,
* дату создания и истечения,
* подпись, чтобы нельзя было подделать.

Сервер проверяет подпись, не обращаясь к базе,
поэтому JWT быстрее и удобнее для масштабных API и фронтендов вроде React.

---

### 🔹 Как это работает

1. Пользователь отправляет `email + password` на `/api/token/`.
2. Сервер проверяет данные и возвращает два токена:

   * **access token** — короткоживущий, используется для запросов;
   * **refresh token** — долговременный, используется для обновления access-токена.
3. React сохраняет эти токены в `localStorage` или `httpOnly` cookie.
4. При каждом запросе React отправляет:

   ```
   Authorization: Bearer <access_token>
   ```
5. Если access-токен истёк — React делает запрос на `/api/token/refresh/`,
   передавая refresh-токен, и получает новый access-токен.

## 6. Авторизация по email вместо username

### 🔹 Проблема

По умолчанию Django использует поле `username` для логина.
Но современные приложения (включая React) почти всегда используют **email**.

### 🔹 Что мы сделали

Мы создали собственный `EmailTokenObtainPairSerializer`,
который наследует `TokenObtainPairSerializer`, но переопределяет поле `username_field = 'email'`.

Также был написан `EmailBackend`, который позволяет Django искать пользователя по email.

### 🔹 Результат

Теперь вход выполняется через:

```json
{
  "email": "test@example.com",
  "password": "123456"
}
```

и сервер возвращает JWT-токены.

React-приложение имитировало регистрацию и логин через **localStorage** —

## 8. Основные принципы безопасности

### 🔹 Почему JWT безопасен

JWT подписывается **секретным ключом сервера**.
Клиент не может его подделать, потому что не знает этот ключ.
Поэтому серверу не нужно хранить токены в базе — он просто проверяет подпись.

### 🔹 Потенциальные уязвимости

* Нельзя хранить токены в `localStorage` на небезопасных сайтах (риск XSS).
* Лучше использовать `httpOnly` cookie, если приложение большое и публичное.
* Access-токены должны иметь короткий срок жизни (например, 5–10 минут).
* Refresh-токены — дольше (несколько дней), чтобы пользователь не логинился заново каждый раз.

## 9. Ошибки, которые мы прошли (и чему они научили)

* **403 Forbidden: Authentication credentials were not provided**
  → Проблема с отсутствием заголовка `Authorization`.
  Мы узнали, что токен нужно передавать явно.

* **Bad Request: This field is required**
  → Неверное имя поля в JSON (React отправлял `username`, а Django ждал `email`).
  Мы осознали важность строгого совпадения схем данных.

* **Unauthorized / 401**
  → JWT недействителен, истёк, или не был передан.
  Это напомнило о важности refresh-токенов и корректной передачи заголовков.

    ↓

    ↓
