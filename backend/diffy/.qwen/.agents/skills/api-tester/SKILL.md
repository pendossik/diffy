# API Tester

## Обзор

Агент для автоматизированного тестирования Django REST API приложений. Проводит комплексное тестирование сервисного слоя и API views, выявляет уязвимости безопасности, отклонения от REST стандартов и логические ошибки.

**Этот скил дополняет rest-api-view-builder** — после того как API написано, этот агент проверяет его корректность.

## Когда использовать этот скил

- ✅ После создания нового API приложения (categories, products, etc.)
- ✅ После значительных изменений в сервисах или views
- ✅ Перед релизом для валидации корректности работы
- ✅ Для поиска уязвимостей и отклонений от стандартов
- ✅ Для документирования покрытия тестами

## Железное правило

**Запрещено редактировать код приложения** — если найдены ошибки, редактируем только тесты, а отклонения заносим в отчёт README.md. Исключение: можно создавать новые фикстуры в `unit/<app>/conftest.py`.

## Основные принципы

1. **Анализ перед тестированием** — изучить модели, сервисы, views, документацию
2. **Покрытие всех сценариев** — позитивные, негативные, граничные случаи, уязвимости
3. **Изоляция тестов** — каждый тест работает со свежей БД, фикстуры с очисткой
4. **Документирование отклонений** — все найденные проблемы заносить в README.md
5. **Наследование стиля** — следовать структуре тестов из categories/products

## Быстрый старт

### 7-шаговый процесс тестирования

**Шаг 1: Ознакомиться с приложением**
Прочитать файлы: `models.py`, `services.py`, `views.py`, `serializers.py`, `urls.py`

**Шаг 2: Изучить примеры тестов**
Открыть `test/unit/categories/test_services.py` и `test/unit/categories/test_views.py`

**Шаг 3: Создать структуру тестов**
```bash
test/unit/<app>/
├── conftest.py       # Локальные фикстуры
├── test_services.py  # Тесты сервисов
├── test_views.py     # Тесты views
└── README.md         # Отчёт о тестировании
```

**Шаг 4: Написать фикстуры**
Создать в `conftest.py`:
- Фикстуры очистки (`cleanup_<model>`)
- Фикстуры тестовых данных (`test_<model>`, `many_<model>`, `searchable_<model>`)
- Фикстуры для специальных сценариев

**Шаг 5: Написать тесты сервисов**
В `test_services.py`:
- Тесты на CRITICAL уязвимости (`_is_admin`, race condition, валидация)
- Тесты на HIGH функциональность (CRUD операции, поиск, фильтрация)
- Тесты на MEDIUM стандарты (граничные случаи, логирование)

**Шаг 6: Написать тесты views**
В `test_views.py`:
- Тесты аутентификации (401 без auth)
- Тесты прав доступа (403 для не-админов)
- Тесты CRUD операций
- Тесты валидации и формата ошибок

**Шаг 7: Запустить тесты и задокументировать**
```bash
python -m pytest test/unit/<app>/ -v
```
Заполнить `README.md` по шаблону из categories.

## Красные флаги — STOP

STOP когда:
- «Нашёл баг в коде — исправлю его» → нельзя, только задокументировать в README.md
- «Скопирую тесты из categories без анализа» → адаптировать под специфику приложения
- «Протестирую только позитивные сценарии» → обязательно негативные и граничные
- «Не буду создавать фикстуры» → без фикстур тесты не изолированы
- «Забуду про документирование отклонений» → README.md обязателен
- «Протестирую только сервисы или только views» → нужно и то, и другое
- «Игнорирую уязвимости безопасности» → проверить аутентификацию, авторизацию, валидацию

**Всё это означает: STOP. Пересмотреть принципы и референсы.**

## Реальное влияние

Из опыта тестирования:
- **Categories**: 98 тестов, 100% pass, 0 отклонений
- **Products**: 106 тестов, 100% pass, 2 отклонения исправлено в процессе
- **Время тестирования**: ~35 секунд на полное покрытие
- **Покрытие**: CRITICAL (23), HIGH (14), MEDIUM (13) уязвимостей

## Типы тестовых сценариев

### 🔴 CRITICAL (обязательно)
- Проверка `_is_admin()` с None, AnonymousUser, неактивными пользователями
- Race condition при создании/обновлении (атомарность проверки и записи)
- Валидация длины полей (max_length)
- Валидация пустых значений

### 🟠 HIGH (обязательно)
- Аутентификация (401 без auth)
- Авторизация (403 для не-админов)
- CRUD операции (Create, Read, Update, Delete)
- Поиск и фильтрация
- Уникальность данных (дубликаты)

### 🟡 MEDIUM (рекомендуется)
- Граничные случаи (спецсимволы, unicode, пробелы)
- Формат ответов об ошибках
- Логирование операций
- Пагинация

### 🟢 LOW (на будущее)
- Производительность (кэширование, N+1 запросы)
- Rate limiting
- Soft delete
