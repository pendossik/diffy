# REST API View Builder

## Обзор

Скил для создания REST API view классов в Django проектах с использованием Django REST Framework и drf-spectacular для Swagger документации.

**Возможности:**
- Создание view классов для CRUD операций
- Документирование API через `@extend_schema` и `@extend_schema_view`
- Настройка маршрутизации в `urls.py`
- Добавление примеров запросов и ответов для Swagger UI

## Когда использовать этот скил

- Создание нового API эндпоинта для модели
- Рефакторинг function-based views в class-based views
- Добавление Swagger документации к существующим views
- Настройка RESTful маршрутов в urls.py
- Унификация стиля API представлений в проекте

## Железное правило

**Проектируй для разработчиков API, а не для сервера.** Каждый эндпоинт должен иметь понятную документацию с примерами использования.

## Основные принципы

1. **Единый класс на ресурс** — объединять операции в одном классе (ListCreateView, DetailView)
2. **Документирование через декораторы** — использовать `@extend_schema_view` для каждого метода
3. **Примеры обязательны** — каждый запрос и ответ должен иметь пример в Swagger
4. **RESTful маршруты** — использовать стандартные HTTP методы вместо `/create/`, `/update/`, `/delete/`
5. **Уникальные operation_id** — явно указывать для каждого эндпоинта

## Быстрый старт

1. **Ознакомиться с моделями и сервисами** — прочитать файлы `models.py`, `services.py`, `serializers.py`
2. **Определить CRUD операции** — какие методы нужны (GET список, GET детали, POST, PUT, PATCH, DELETE)
3. **Создать view классы** — использовать `@extend_schema_view` с документацией для каждого метода
4. **Настроить urls.py** — подключить view классы с RESTful маршрутами
5. **Добавить примеры** — указать `OpenApiExample` для запросов и ответов

## Ключевые напоминания

- **Использовать `@extend_schema_view`** — один декоратор на класс с методами get/post/put/patch/delete
- **Указывать `operation_id`** — уникальный идентификатор для каждого эндпоинта (например, `categories_list`, `categories_create`)
- **Добавлять примеры** — минимум один пример запроса и ответа для каждого метода
- **Документировать ошибки** — указывать 400/401/403/404/409 ответы с примерами
- **RESTful маршруты** — `GET/POST /`, `GET/PUT/PATCH/DELETE /<id>/`
- **Использовать `id` в документации** — параметр path должен называться `id` (не `pk`) для соответствия с URL паттерном
- **Imperative voice** — документация в повелительном наклонении

## Красные флаги — STOP

STOP когда:
- «Создам отдельный view для каждого метода» → объединить в один класс
- «Не буду добавлять примеры для Swagger» → добавить минимум по одному примеру
- «Использую `/create/` и `/update/` в urls» → заменить на RESTful методы
- «Не укажу operation_id» → явно прописать для каждого эндпоинта
- «Документирование потом» → документировать сразу при создании
- «Скопирую документацию из другого view» → адаптировать под конкретный эндпоинт
- «Не обработаю ошибки» → указать все возможные коды ответов
- «Использую `name='pk'` в OpenApiParameter» → использовать `name='id'` для соответствия с URL

**Всё это означает: STOP. Пересмотреть принципы и референсы.**
