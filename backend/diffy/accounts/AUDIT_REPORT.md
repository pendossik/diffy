## 🔴 CRITICAL уязвимости

### 1. Отсутствие проверки `user.is_authenticated` в `get_user_profile()`

**Файл:** `services.py`, метод `get_user_profile()`  
**Проблема:** Метод принимает объект `User` без проверки на аутентификацию.

```python
# Уязвимый код
@staticmethod
def get_user_profile(user: User) -> dict:
    return {
        'email': user.email,
        'role': user.role,
        'is_active': user.is_active,
        'date_joined': user.date_joined
    }
```

**Риск:** Передача неаутентифицированного пользователя может вызвать AttributeError  
**Рекомендация:** Добавить проверку `if not user or not user.is_authenticated: raise PermissionError`

---

### 2. Race condition при регистрации (дублирование email)

**Файл:** `services.py`, метод `register_user()`  
**Проблема:** Проверка `User.objects.filter(email=email).exists()` и создание не атомарны.

```python
# Уязвимый код
if User.objects.filter(email=email).exists():  # Проверка
    raise ValueError("Email уже зарегистрирован")
user = User.objects.create_user(email=email, ...)  # Создание (не атомарно!)
```

**Риск:** Два одновременных запроса могут создать двух пользователей с одним email  
**Рекомендация:** Использовать `transaction.atomic()` с `select_for_update()` или уникальное ограничение на уровне БД

---

### 3. Отправка токена активации в открытом виде в URL

**Файл:** `services.py`, метод `register_user()`  
**Проблема:** Токен активации передаётся в URL, который логируется прокси-серверами.

```python
activation_link = f'{settings.BACKEND_URL}/api/accounts/activate/{activation_token}/'
```

**Риск:** Токен может быть перехвачен из логов прокси/балансировщика  
**Рекомендация:** Передавать токен через POST запрос с телом или использовать одноразовые ссылки

## 🟠 HIGH уязвимости

### 4. Утечка информации при регистрации

**Файл:** `services.py`, `register_user()`  
**Проблема:** Разные сообщения об ошибке для существующего email и слабого пароля.

```python
logger.warning(f"Регистрация не удалась: Email {email} уже зарегистрирован.")
raise ValueError("Email уже зарегистрирован")

logger.warning(f"Регистрация: слабый пароль для {email}")
raise ValueError("Пароль не соответствует требованиям безопасности")
```

**Риск:** Атакующий может определить, зарегистрирован ли email  
**Рекомендация:** Использовать универсальное сообщение "Ошибка регистрации"

---

### 5. Отсутствие rate limiting для регистрации

**Файл:** `views.py`, `RegisterView`  
**Проблема:** Нет ограничений на частоту запросов регистрации.

**Риск:** Возможна спам-атака или перебор email  
**Рекомендация:** Добавить `throttle_classes = [UserRateThrottle]`

---

### 6. Логирование email в открытом виде

**Файл:** `services.py`,多处  
**Проблема:** Email пользователей логируется в открытом виде.

```python
logger.warning(f"Регистрация не удалась: Email {email} уже зарегистрирован.")
logger.info(f"Письмо с активацией отправлено на {email}")
```

**Риск:** Утечка персональных данных через логи  
**Рекомендация:** Использовать маскирование: `user***@example.com`

---

### 7. Неполная валидация пароля

**Файл:** `services.py`, `register_user()`  
**Проблема:** `validate_password()` вызывается без передачи объекта пользователя.

```python
validate_password(password, user=None)  # ← None вместо user
```

**Риск:** Не все валидаторы работают корректно без объекта user  
**Рекомендация:** Создавать временный объект user для валидации

---

### 8. Возврат user_id в JWT ответе

**Файл:** `services.py`, `authenticate_user()`  
**Проблема:** В ответе возвращается `user_id` отдельно от токенов.

```python
return {
    'refresh': str(refresh),
    'access': str(refresh.access_token),
    'user_id': user.id,  # ← Избыточно
    'email': user.email,
    'role': user.role
}
```

**Риск:** Раскрытие внутренней структуры ID пользователей  
**Рекомендация:** Удалить `user_id` из ответа, использовать только токены

---

## 🟡 MEDIUM уязвимости

### 9. Отсутствие верификации email после активации

**Файл:** `services.py`, `activate_account()`  
**Проблема:** Нет проверки, что email действительно принадлежит пользователю.

**Риск:** Активация чужого аккаунта при утечке токена  
**Рекомендация:** Добавить дополнительную верификацию (SMS, 2FA)

---

### 10. Неполная документация Swagger

**Файл:** `views.py`, `@extend_schema`  
**Проблема:** Не все эндпоинты имеют полную документацию ошибок (401, 403, 404).

**Риск:** Разработчики фронтенда не знают все возможные ошибки  
**Рекомендация:** Добавить `responses={401: ..., 403: ..., 404: ...}` во все декораторы

---

### 11. Неявная зависимость от `token_blacklist`

**Файл:** `services.py`, `logout_user()`  
**Проблема:** Метод требует установленное приложение `token_blacklist`, но нет проверки.

**Риск:** Ошибка при отсутствии приложения в INSTALLED_APPS  
**Рекомендация:** Добавить проверку `if 'token_blacklist' in settings.INSTALLED_APPS`

---

### 12. Отсутствие проверки сложности пароля

**Файл:** `serializers.py`, `RegisterSerializer`  
**Проблема:** Только `min_length=8`, нет проверки на сложность.

```python
password = serializers.CharField(write_only=True, min_length=8)
```

**Риск:** Пользователи могут устанавливать слабые пароли  
**Рекомендация:** Добавить валидаторы сложности (цифры, спецсимволы, заглавные буквы)
